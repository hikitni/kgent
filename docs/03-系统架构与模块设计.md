# 03 系统架构与模块设计

- 项目名称：**Kgent V3**
- 文档版本：v3.1
- 更新日期：2026-03-04

## 1. 总体架构

系统采用 **Python Package 分层架构**，入口为 `main.py`，核心逻辑封装于 `daily_reporter/` 包，UI 层支持交互式终端菜单（基于 `rich`）。采集机制为**快照对比**，不强依赖进程持续运行。

**目录结构**

```
d:\AiGithub\
  main.py                    # 程序入口
  config.json                # 运行时配置
  config.example.json        # 配置模板
  requirements.txt           # 依赖声明
  daily_reporter/            # 核心 Package
    __init__.py
    config.py                # 配置加载
    snapshot.py              # 快照采集与管理
    diff.py                  # 快照对比（含行级 diff）
    reporter.py              # 日报生成（含 diff 预览章节）
    ai.py                    # 多提供商 AI 智能总结
    tasks.py                 # Windows 计划任务管理（schtasks.exe 封装）
    ui.py                    # 交互式终端界面
  .daily_reporter/           # 快照数据存储（运行时生成）
    index.json
    snapshots/
  reports/                   # 日报输出（运行时生成）
  docs/                      # 项目文档
```

**三条主流程**

```
snapshot  扫描监控目录 → 计算哈希与行列表 → 写快照文件 → 更新索引
report    选两个快照  → 逐文件对比差异  → 聚合统计  → 生成 Markdown → 可选 AI 总结
watch     监听时钟    → 到达配置时间    → 自动触发 snapshot
scheduled Windows 计划任务 → 到达时间点     → python main.py snapshot → 进程退出
```

**核心优势**：快照是独立 JSON 文件，电脑重启或 watch 未运行均不丢失状态，日报仅由两个快照点的差量决定。

## 3. 模块设计

### 3.1 config.py — 配置模块

- 输入：`config.json`
- 输出：`AppConfig` 数据类

关键字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `watch_paths` | List[Path] | 监控目录列表（自动 resolve 绝对路径）|
| `ignore_dirs` | Set[str] | 忽略目录名 |
| `ignore_suffixes` | Set[str] | 忽略后缀 |
| `ignore_patterns` | List[str] | 路径正则忽略 |
| `snapshot_root` | Path | 快照存储根目录 |
| `output_root` | Path | 日报输出目录 |
| `auto_snapshot_times` | List[str] | 定时快照时间，如 `["06:00","18:00"]` |
| `max_file_size_kb` | int | 大文件阈值，超过只存哈希 |
| `ai_provider` | str | AI 提供商：`disabled` \| `zhipuai` \| `openai` \| `ollama` |
| `ai_model` | str | 模型名称（如 `glm-4-flash`、`deepseek-chat`）|
| `ai_api_key` | str | API 密钥（Ollama 可为空）|
| `ai_base_url` | str | 自定义端点，OpenAI 兼容接口或 Ollama 地址 |
| `ai_prompt_template` | Path | 报告生成 prompt 模板文件路径（相对于 config.json，留空则用内置默认模板）|

### 3.2 snapshot.py — 快照模块

**快照文件结构**（`snapshot_root/snapshots/snapshot-YYYYMMDD-HHmmss.json`）

```json
{
  "id": "20260304-060000",
  "timestamp": "2026-03-04T06:00:00",
  "label": "morning",
  "trigger": "auto",
  "file_count": 42,
  "files": {
    "D:/project/main.py": {
      "hash": "a1b2c3d4...",
      "lines": ["line1", "line2"]
    }
  }
}
```

**快照索引**（`snapshot_root/index.json`）— 仅存元数据，不含文件内容

```json
[
  {"id": "20260304-060000", "timestamp": "...", "label": "morning", "trigger": "auto", "file_count": 42}
]
```

**公开 API**

| 函数 | 说明 |
|---|---|
| `take_snapshot(cfg, label, trigger)` | 扫描目录，写快照文件，更新索引，返回快照 dict |
| `load_snapshot(cfg, snap_id)` | 加载指定快照 JSON |
| `load_index(cfg)` | 读取 index.json |
| `save_index(cfg, index)` | 写入 index.json |
| `delete_snapshot(cfg, snap_id)` | 删除快照文件并从索引移除 |

**关键流程**：
1. 遍历 `watch_paths`，对每个文件检查 ignore 规则
2. 文件大小 ≤ `max_file_size_kb`：读取文本，存行列表 + MD5
3. 文件大小 > `max_file_size_kb`：读字节，只存 MD5，`lines=null`
4. 写入快照 JSON，更新 `index.json`

### 3.3 diff.py — 对比模块

- 输入：快照 A（起点）、快照 B（终点）
- 输出：变更记录列表 + 摘要字典

**对比规则**

| 情况 | 状态 |
|---|---|
| 仅在 B 中存在 | `created` |
| 仅在 A 中存在 | `deleted` |
| 两者 hash 相同 | 跳过（unchanged）|
| 两者 hash 不同 | `modified` |

**行级统计**

$$added = |B.lines \setminus A.lines|, \quad removed = |A.lines \setminus B.lines|, \quad net = added - removed$$

**diff_snapshots 返回列表的单条结构**

| 字段 | 类型 | 说明 |
|---|---|---|
| `path` | str | 文件绝对路径 |
| `status` | str | `created` / `modified` / `deleted` |
| `added_lines` | int | 新增行数 |
| `removed_lines` | int | 删除行数 |
| `net_lines` | int | 净行变化 |
| `ext` | str | 文件后缀 |
| `parent` | str | 父目录名 |
| `diff_content` | List[str] | 行级 unified diff（`+`/`-` 行，最多 60 行）|
| `large_file` | bool | `True` 表示文件超过 `max_file_size_kb`，无行内容 |

**公开 API**

| 函数 | 说明 |
|---|---|
| `diff_snapshots(snap_a, snap_b)` | 返回变更文件列表（含以上全部字段）|
| `summarize_diff(diffs)` | 聚合统计：总数/各状态/行计数/Top-ext/Top-dir |

### 3.4 reporter.py — 日报模块

- 输入：快照 A、快照 B
- 输出：`output_root/report-{A.id}-to-{B.id}.md`

**公开 API**

| 函数 | 说明 |
|---|---|
| `generate_report(cfg, snap_a, snap_b)` | 生成 Markdown 日报，返回输出路径 |
| `list_reports(cfg)` | 扫描 output_root，返回按修改时间降序的日报列表（含文件名/路径/大小/时间/起终点快照 ID）|
| `delete_report(path)` | 删除指定日报文件，返回是否成功 |

**日报章节结构（8章）**

| 章节 | 内容 |
|---|---|
| 一 | 概览（时间区间、快照 ID、变更总数）|
| 二 | 变更文件类型分布 Top5 |
| 三 | 高频变更目录 Top5 |
| 四 | 🟢 新增文件明细（最多 20 条）|
| 五 | 🟡 修改文件明细（最多 20 条）|
| 六 | 🔴 删除文件明细（最多 20 条）|
| **七** | **变更内容预览**（per-file ` ``` diff ` 块，最多60行/文件）|
| 八 | 工作总结草稿（含占位符）|

> 若 `ai.append_ai_summary()` 被调用，日报末尾还会追加 **九、AI 智能总结** 章节。

### 3.5 ui.py — 交互式终端界面

基于 `rich` 库实现，每次回到主菜单自动重载 `config.json`（编辑配置后无需重启）。

**主菜单**

| 键 | 功能 | 说明 |
|---|---|---|
| `1` | 📸 打快照 | 手动触发，可自定义 label |
| `2` | 📋 浏览/管理快照 | 列表 + 详情（文件级预览）+ 删除 |
| `3` | 🔍 对比快照 | 任选两点，展示变更摘要与文件明细 |
| `4` | 📄 生成日报 | 选区间 → 可选自动打日结快照 → 生成 Markdown → 可选打开 → **可选 AI 总结** |
| `5` | 📑 日报管理 | 列出全部日报，支持打开、删除、**AI 总结** |
| `6` | ⏱ Watch | 按 `auto_snapshot_times` 每 30s 检查定时触发（小提示：推荐用菜单 8 替代）|
| `7` | ⚙️ 查看配置 | 分组展示 config.json 全字段 + 可用系统编辑器打开 |
| `8` | 📅 **定时任务管理** | 查看/注册/删除 Windows 计划任务，立即触发快照 |
| `0` | 🚪 退出 | — |

**日报生成流程**

```
选择起点快照 A
→ 是否用默认区间（最后两个快照）
→ 是否自动打[日结快照]作为终点
→ 生成 Markdown 日报
→ 是否立即用系统默认程序打开文件
→ 是否使用 AI 对日报进行智能总结？（ai_provider != disabled 时出现）
```

**快照管理操作**（浏览页）

- `[d]` 查看快照详情：展示文件列表（前 30 条）+ 哈希 + 行数
- `[x]` 删除快照：确认后删除文件并移除索引

**日报管理操作**（菜单 5）

| 输入 | 操作 |
|---|---|
| `序号`（如 `1`）| 用系统默认程序打开日报 |
| `d+序号`（如 `d1`）| 删除日报文件（二次确认）|
| `s+序号`（如 `s1`）| 对该日报调用 AI 智能总结，追加「九、AI 智能总结」章节 |
| `0` | 返回主菜单 |

### 3.6 tasks.py — Windows 计划任务管理

- 输入：`AppConfig` + `config_path`
- 输出：封装 `schtasks.exe` 的搜索、注册、删除操作

**设计思路**：将“定时触发”的责任全部转移给操作系统。每个时间点对应一条计划任务，事件到达时由 Windows 自行调用 `python main.py snapshot`，程序完成快照后自行退出，无需任何进程持续运行。

**任务命名规则**：`DailyReporter-HHMM`（如 `DailyReporter-0600`）

**公开 API**

| 函数 | 说明 |
|---|---|
| `list_tasks()` | 查询所有 `DailyReporter-*` 任务，返回名称/时间/下次运行/状态 |
| `register_tasks(cfg, config_path)` | 按 `auto_snapshot_times` 创建/覆盖任务；删除已从配置移除的旧任务 |
| `delete_task(task_name)` | 删除单条任务 |
| `delete_all_tasks()` | 删除全部 `DailyReporter-*` 任务 |
| `run_task_now(cfg, hm)` | 立即触发快照（直接调用 `take_snapshot`，不经 schtasks）|

### 3.7 ai.py — AI 智能总结模块

- 输入：`AppConfig`（含 AI 配置字段）+ 日报文本
- 输出：AI 生成的中文总结字符串；可追加写入 Markdown 文件

**提供商抽象**

```
BaseProvider (ABC)
  ├── ZhipuAIProvider   # 智谱 AI GLM 系列（pip install zhipuai）
  ├── OpenAIProvider    # OpenAI / DeepSeek / Moonshot 等兼容接口（pip install openai）
  └── OllamaProvider    # 本地 Ollama，纯 HTTP，无需额外安装
```

**公开 API**

| 函数 | 签名 | 说明 |
|---|---|---|
| `get_provider(cfg)` | `(AppConfig) → BaseProvider` | 工厂函数，按 `cfg.ai_provider` 返回对应提供商实例 |
| `generate_full_report(cfg, raw_report)` | `(AppConfig, str) → str` | 全文重写：将机器版 Markdown 投送 AI，返回 AI 出工中文日报 |
| `summarize_report(cfg, text)` | `(AppConfig, str) → str` | 追加型总结（圈选功能，保留兼容）|
| `append_ai_summary(path, text, name)` | `(Path, str, str) → None` | 追加「九、AI 智能总结」章节到 Markdown 文件 |
| `build_prompt(report_text)` | `(str) → str` | 构造中文 PM 助手 prompt，超 10000 字符自动截断 |

**配置字段**（`config.json`）

```json
"ai_provider": "disabled",   // disabled | zhipuai | openai | ollama
"ai_model":    "",            // 如 glm-4-flash、deepseek-chat、llama3
"ai_api_key":  "",            // Ollama 可为空
"ai_base_url": ""             // 自定义端点（Ollama 默认 http://localhost:11434）
"ai_prompt_template": "ai_report_prompt.md"  // prompt 模板文件（用户可自由编辑）
```

**设计要点**：AI 为可选功能。配置 AI 后日报生成流程：
1. 首先生成机器版 Markdown（含完整统计数据和 diff 内容）
2. 将晳机版内容投送给 AI，结合 `ai_report_prompt.md` 模板生成叙述性日报
3. AI 输出覆盖原始文件（失败时保留机器版）
4. `ai_provider="disabled"`（默认）时不导入任何 AI SDK

### 3.8 Watch 调度（呭起方案）

- 每 30 秒检查一次系统时钟
- 当前时间（HH:MM）命中 `auto_snapshot_times` 且当天未触发 → 调用 `take_snapshot`
- 00:00 重置当天已触发记录
- 支持 Ctrl+C 安全退出
